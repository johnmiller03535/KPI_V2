## 1. ИСТОЧНИКИ ДАННЫХ ИЗ REDMINE

### Что использовать

| Поле | Где | ID | Использование |
|---|---|---|---|
| `manager` | Карточка пользователя `/people/{id}` | — | **Прямой руководитель** — основа матрицы подчинённости |
| `КПИ_Номер` | CF пользователя | CF 211 | `pos_id` сотрудника |
| `Идентификатор должности` | CF задачи | CF 206 | `role_id` (ЦТР_НАЧ_071 и т.д.) |
| `Подразделение (аббревиатура)` | CF пользователя | CF 124 | Принадлежность к подразделению |
| `Дата начала работы` | Карточка пользователя | — | Дата приёма |
| `Дата увольнения` | Карточка пользователя | — | Дата увольнения |
| `Telegram id` | CF пользователя | CF 3 | Telegram chat_id |

### Что НЕ использовать

| Поле | Причина |
|---|---|
| CF 4 «Соисполнитель» | Не поддерживается, содержит устаревшие данные, нет валидации |
| CF 14 «Вакансия» | Заполняется непоследовательно, функционал вакансий в Redmine не реализован |
| Группы Redmine | Создаются вручную, нет гарантии актуальности |
| Проекты kpi-2024, kpi-2025 | Устаревшие. Работаем только с проектом `kpi-reports` |

### Фильтрация активных сотрудников

Из Redmine брать только пользователей у которых:
- Статус = **active** (не `locked`, не `registered`)
- Подразделение = **ГКУ "РЦТ"** (поле Подразделение в карточке)
- НЕ брать пользователей со статусом `registered` — они не активны

## 2. АЛГОРИТМ СИНХРОНИЗАЦИИ ПОДЧИНЁННОСТИ

```
Для каждого активного пользователя из /people (ГКУ "РЦТ", статус active):

1. pos_id          = CF_211 (КПИ_Номер)
2. role_id         = KPI_Mapping.get_role_id(pos_id)
3. department      = CF_124 (Подразделение аббревиатура)
4. telegram_id     = CF_3
5. manager_id      = поле "Менеджер" (user_id прямого руководителя)
6. hire_date       = Дата начала работы
7. fire_date       = Дата увольнения

Если manager_id задан:
  manager_user = /people/{manager_id}
  manager_pos_id = manager_user.CF_211
  manager_role_id = KPI_Mapping.get_role_id(manager_pos_id)
  subordination[role_id] = manager_role_id

Если fire_date <= today → статус = dismissed
```

### Обнаружение изменений (ежедневно в 09:00)

Сравниваем текущие данные Redmine с таблицей `employees` в БД:

| Событие | Признак | Действие |
|---|---|---|
| Новый сотрудник | есть в Redmine, нет в employees | Создать запись, уведомить admin |
| Уволен | `status=locked` или `fire_date <= today` | `employees.status = dismissed`, уведомить admin |
| Смена должности | изменился CF_211 или CF_206 | Обновить position_id, уведомить admin |
| Смена руководителя | изменился `manager_id` | Обновить subordination, уведомить admin |
| Появился без pos_id | CF_211 пустой | Уведомить admin: «назначьте должность» |

## 3. СМЕНА РУКОВОДИТЕЛЯ В СЕРЕДИНЕ ПЕРИОДА

### Решение: два отчёта, автоматически

При обнаружении смены manager_id у сотрудника в активном периоде:

1. Система **автоматически** создаёт новую задачу в Redmine (проект `kpi-reports`)
2. Создаётся новый `KpiSubmission` со статусом `draft` — новому руководителю
3. Старый submission получает статус `transferred` (закрыт на дату перевода)
4. Администратор получает Telegram-уведомление: «Сотрудник X переведён. Автоматически создан второй отчёт»

### Оценка при двух отчётах

Оценка производится по задачам каждого периода независимо — финансисты учитывают отработанные дни отдельно. В системе KPI-портала два отчёта оцениваются независимо каждым своим руководителем.

## 4. ЗАМЕЩЕНИЕ

### Правила

- Замещение назначает **сам руководитель** (не администратор)
- Заместитель — **только из своей ветки подчинения**
  - Для начальника управления → его зам. нач. управления
  - Для замдиректора → другой замдиректор
  - Нельзя назначить произвольного сотрудника из другого подразделения
- Замещение имеет **дату начала и дату окончания**
- Заместитель **не может делегировать** замещение дальше
- Отчёты, утверждённые заместителем, **помечаются**: `approved_by_deputy = true`
- Подчинённых **не уведомляем** о замещении
- Подчинённых **уведомляем** в Telegram при назначении замещения (кто их будет проверять)

### Таблица active_deputies

```sql
CREATE TABLE active_deputies (
    id SERIAL PRIMARY KEY,
    manager_role_id VARCHAR NOT NULL,   -- кто в отпуске
    deputy_role_id VARCHAR NOT NULL,    -- кто замещает (из той же ветки)
    starts_at DATE NOT NULL,
    ends_at DATE NOT NULL,
    created_by INTEGER,                 -- redmine_id руководителя
    is_active BOOLEAN DEFAULT TRUE
);
```

## 5. РОЛИ В СИСТЕМЕ

| Роль | Кто | Права |
|---|---|---|
| `admin` | Системный администратор | Полный доступ, редактирование матрицы |
| `hr` | HR-специалист | Структура подразделений, движение кадров, просмотр матрицы |
| `manager` | Руководитель | Видит своих подчинённых, утверждает отчёты, назначает замещение |
| `employee` | Сотрудник | Заполняет KPI-форму |
| `finance` | Финансовый блок | Просмотр утверждённых отчётов |

### Роль HR (новая)

HR видит и может:
- Просматривать всю матрицу подчинённости
- Видеть список новых/уволенных/переведённых сотрудников
- Инициировать запрос администратору на изменение матрицы
- Просматривать историю изменений подчинённости
- Видеть сотрудников без должности / без руководителя

HR **не может**:
- Редактировать матрицу самостоятельно (только через запрос admin)
- Видеть содержимое KPI-отчётов

## 6. УПРАВЛЕНИЕ KPI-КАРТОЧКАМИ (новый раздел)

### Что такое KPI-карточка

Карточка оценки = набор KPI-показателей для конкретной должности из `KPI_Mapping.xlsx`.

### Функционал

| Действие | Кто | Где |
|---|---|---|
| Просмотр карточки по должности | admin, hr | /admin → таб «KPI-карточки» |
| Проверка соответствия карточки должности | admin | Сверка с xlsx |
| Загрузка новой версии KPI_Mapping.xlsx | admin | /admin → загрузка файла |
| Обновление карточки для конкретной должности | admin | Редактирование показателей |
| Просмотр общего списка карточек | admin, hr | Таблица всех 91 должности |

### Приоритет реализации

Сначала: просмотр и проверка соответствия.
Потом: загрузка нового xlsx и обновление карточек.

## 7. ИНТЕРФЕЙС

### Admin-панель /admin — табы

1. Обзор
2. Периоды
3. Сотрудники
4. Подчинённость ← новый
5. KPI-карточки ← новый
6. Аудит
7. Уведомления
8. Диагностика ← уже реализован (data-health)

### Таб «Подчинённость»

- Таблица: Должность / Сотрудник / Подразделение / Руководитель / Статус
- Статусы: ✅ ок / ⚠️ нет руководителя / ❌ руководитель не найден / 👤 вакансия
- Фильтр по подразделению
- Входящие запросы от руководителей об ошибках
- Кнопка «Изменить руководителя» → dropdown из активных сотрудников той же ветки

### Таб «KPI-карточки»

- Dropdown: выбор должности (все 91)
- Отображение карточки: показатели, формулы, веса, тип (binary_auto/manual/numeric)
- Статус: ✅ карточка есть и сумма весов = 100% / ⚠️ проблема
- Кнопка «Загрузить новый KPI_Mapping.xlsx»
