# Бизнес-логика: Подчинённость
> Статус: черновик | Дата: 2026-04-19 | Требует подтверждения

---

## 1. МОДЕЛЬ ПОДЧИНЁННОСТИ

### Что такое подчинённость в системе
Матрица подчинённости определяет **кто кому подчиняется** для целей KPI-отчётности:
- Какой руководитель утверждает отчёт сотрудника
- Кто оценивает binary_manual KPI сотрудника
- Кто видит отчёты в разделе /review

### Текущая структура (subordination.json)
```
evaluator:    role_id → role_id руководителя (кто утверждает отчёт)
deputy_for:   role_id → role_id заместителя (кто замещает при отсутствии)
ruk_assignments: замдиректора → список начальников управлений
```

### Иерархия (зафиксировано)
```
Замдиректора
  └── Начальник управления
        ├── Зам. нач. управления — нач. отдела
        │     └── Сотрудники отдела
        └── Зам. нач. управления — нач. отдела
              └── Сотрудники отдела
```

## 2. РОЛИ И ПРАВА

### Администратор
- Видит полную матрицу подчинённости всей организации
- Редактирует матрицу (назначает/меняет руководителя для любой должности)
- Получает уведомления об ошибках подчинённости от руководителей
- Запускает синхронизацию с Redmine вручную
- Видит список новых/изменившихся сотрудников после синхронизации

### Руководитель
- Видит **только свою цепочку** подчинённых (не всю организацию)
- НЕ может редактировать матрицу самостоятельно
- Может **отправить запрос администратору** на исправление ошибки
  - Например: «Иванов не должен быть в моём списке» или «Петров отсутствует в моём списке»
- Может назначить **замещающего** на период своего отсутствия

### Сотрудник
- Не видит матрицу подчинённости
- Видит только имя своего руководителя в профиле/форме

## 3. СИНХРОНИЗАЦИЯ С REDMINE

### Что синхронизируется ежедневно (09:00 МСК, APScheduler)
1. **Новые сотрудники** — появились в Redmine, нет в таблице employees
2. **Уволенные** — есть в employees, нет в Redmine (или статус locked)
3. **Смена должности** — изменилось кастомное поле CF_ROLE_ID=206
4. **Смена подразделения** — изменился проект сотрудника в Redmine

### Что НЕ синхронизируется автоматически
- Матрица подчинённости (evaluator) — настраивается администратором вручную
- Замещение (deputy) — назначается руководителем вручную

### Действия при обнаружении изменений
| Событие | Автоматическое действие | Уведомление |
|---|---|---|
| Новый сотрудник | Создать запись employees, position_id = null | Администратору: «Новый сотрудник, назначьте должность и руководителя» |
| Уволен | Статус = dismissed, submissions блокируются | Администратору: «Сотрудник уволен» |
| Смена должности | Обновить position_id | Администратору: «Изменилась должность, проверьте подчинённость» |
| Смена подразделения | Обновить department_code | Администратору: «Сотрудник переведён, настройте подчинённость» |

### ❓ ОТКРЫТЫЙ ВОПРОС — Что хранит Redmine о структуре
Необходимо изучить Redmine API и ответить:
- Есть ли в Redmine группы/отделы (Groups)?
- Как определяется принадлежность к подразделению — через проект kpi-* или отдельное поле?
- Хранится ли в Redmine информация о прямом руководителе сотрудника?
- Есть ли поле «дата приёма/увольнения»?

**До получения ответов — синхронизация работает как сейчас:** читает CF_ROLE_ID=206 и список пользователей.

## 4. ЗАМЕЩЕНИЕ (ВРЕМЕННОЕ)

### Сценарий
Руководитель уходит в отпуск. На это время его обязанности по утверждению KPI-отчётов берёт другой сотрудник.

### Правила замещения
- Замещение назначает **сам руководитель** (не администратор)
- Один активный заместитель в один момент времени
- Замещение имеет **дату начала и дату окончания**
- Заместитель видит отчёты подчинённых замещаемого руководителя так же, как оригинальный руководитель
- Заместитель может быть любым сотрудником организации (не обязательно из той же цепочки)
- После окончания замещения — права возвращаются оригинальному руководителю

### Кто может быть заместителем
По умолчанию — заместитель из `deputy_for` в subordination.json.
Руководитель может указать другого сотрудника (свободный выбор).

### ❓ ОТКРЫТЫЕ ВОПРОСЫ
- Может ли заместитель сам назначить замещение (делегировать дальше)? → **Предлагаю: нет**
- Что происходит с отчётами, утверждёнными заместителем — они помечаются как «утверждено замещающим»?
- Нужно ли уведомлять подчинённых о том, что их руководитель сейчас заменён?

## 5. СМЕНА РУКОВОДИТЕЛЯ В СЕРЕДИНЕ ПЕРИОДА

### Сценарий
Сотрудник переведён в другое подразделение (или сменился руководитель) во время активного отчётного периода.

### Решение: два отчёта
- Текущий submission **закрывается** на дату перевода (частичный период)
- Создаётся **новый submission** от даты перевода до конца периода
- Каждый отчёт утверждается своим руководителем (старым и новым соответственно)
- Итоговая оценка за период = взвешенное среднее двух отчётов (пропорционально дням)

### ❓ ОТКРЫТЫЕ ВОПРОСЫ
- Кто инициирует разбивку на два отчёта — система автоматически или администратор вручную?
- Как считать вес: если сотрудник 20 дней у старого руководителя и 10 дней у нового — оценка 67% первого + 33% второго?
- Что делать с трудозатратами в Redmine — AI должен анализировать только трудозатраты за свой подпериод?
- Нужно ли это реализовывать сейчас, или отложить как edge case?

**Решено:** администратор вручную создаёт второй submission, первый закрывает через новый статус `transferred`.
- При ежедневной синхронизации система **автоматически уведомляет администратора** если у сотрудника изменилась должность или подразделение
- Администратор получает Telegram-сообщение: «Сотрудник X переведён. Возможно, требуется разбивка отчёта»
- Автоматическое создание двух submissions — следующий этап (не сейчас)

## 6. ИНТЕРФЕЙС АДМИНИСТРАТОРА — ПОДЧИНЁННОСТЬ

### Страница /admin → таб «Подчинённость»

#### Отображение
- Таблица: Должность / Подразделение / ФИО (если назначен) / Руководитель / Статус
- Статусы строки: ✅ Настроено / ⚠️ Нет руководителя / ❌ Руководитель не найден в системе
- Фильтр по подразделению
- Поиск по ФИО или должности

#### Редактирование
- Клик на строку → открывается модальное окно
- Dropdown: выбор руководителя из списка всех должностей
- Кнопка «Сохранить» → немедленно применяется

#### Запросы от руководителей
- Раздел «Входящие запросы» — список обращений от руководителей об ошибках
- Каждый запрос: кто прислал / что не так / кнопки «Исправить» и «Отклонить»

### Страница руководителя — просмотр подчинённых
- В /dashboard или отдельная страница /my-team
- Список подчинённых: ФИО / Должность / Статус отчёта за текущий период
- Кнопка «Сообщить об ошибке» → форма с текстом → уходит администратору

## 7. ХРАНЕНИЕ ДАННЫХ

### Текущее состояние
Подчинённость хранится в `reference/subordination.json` — файл в репозитории, read-only в контейнере.

### Целевое состояние (Шаг H)
Таблица `subordination` в PostgreSQL:
```sql
CREATE TABLE subordination (
    id SERIAL PRIMARY KEY,
    role_id VARCHAR NOT NULL UNIQUE,      -- подчинённый (role_id из KPI_Mapping)
    evaluator_role_id VARCHAR,            -- руководитель (может быть NULL)
    deputy_role_id VARCHAR,               -- заместитель по умолчанию
    updated_at TIMESTAMP,
    updated_by INTEGER                    -- redmine_id администратора
);

CREATE TABLE active_deputies (
    id SERIAL PRIMARY KEY,
    manager_role_id VARCHAR NOT NULL,     -- кто в отпуске
    deputy_role_id VARCHAR NOT NULL,      -- кто замещает
    starts_at DATE NOT NULL,
    ends_at DATE NOT NULL,
    created_by INTEGER,                   -- redmine_id руководителя
    is_active BOOLEAN DEFAULT TRUE
);
```

### Переходный период
До реализации Шага H — читать из subordination.json, но дать возможность **просматривать** через UI. Редактирование через UI → пишем обратно в json (временно).
