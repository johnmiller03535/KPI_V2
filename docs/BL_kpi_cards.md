# Бизнес-логика: KPI-карточки
> Версия: 1.0 | Дата: 2026-04-19 | Статус: подтверждено

---

## 1. ЧТО ТАКОЕ KPI-КАРТОЧКА

KPI-карточка — набор показателей оценки для конкретной должности (`pos_id`).
Хранится в `KPI_Mapping.xlsx`, лист `KPI_Indicators`.

Карточка привязана к **должности (pos_id)**, не к сотруднику.
Несколько сотрудников могут иметь одинаковый `pos_id` → одну карточку.
Если у двух «Консультантов» разные карточки — у них **разные pos_id** (разные номера в xlsx).
Индивидуального переопределения карточки для конкретного сотрудника нет.

---

## 2. СТРУКТУРА КАРТОЧКИ

```
pos_id        — числовой идентификатор должности (1–91)
role_id       — строковый код (ЦТР_КОН_075)
role_name     — название должности
unit          — подразделение
indicators[]  — список показателей:
  - indicator     — название показателя
  - criterion     — критерий оценки
  - plan_value    — плановое значение
  - weight        — вес (сумма по карточке = 100%)
  - is_common     — общий показатель (дисциплина, распорядок)
  - formula_type  — тип: binary_auto / binary_manual / threshold /
                         multi_threshold / quarterly_threshold
  - formula_desc  — описание формулы
  - thresholds    — пороговые значения
  - cumulative    — нарастающим итогом (bool)
```

**Инвариант:** сумма `weight` по всем показателям карточки = **100%**.
Карточка невалидна если сумма ≠ 100%.

---

## 3. ЖИЗНЕННЫЙ ЦИКЛ КАРТОЧКИ

### Когда меняются карточки
- **Раз в год** — плановое утверждение показателей на новый год
- **По факту** — при приёме нового сотрудника на новую должность или изменении функций

### Кто инициирует изменение
- **HR** — инициирует: предлагает изменения в карточке
- **Admin** — проверяет корректность (сумма весов, типы формул) и акцептует

### Когда вступают в силу
Изменения вступают в силу **со следующего отчётного периода**.
Текущий активный период работает со старой версией карточки.

---

## 4. ВЕРСИОНИРОВАНИЕ КАРТОЧЕК

### Почему важно
При формировании годового отчёта нужно учитывать обе карточки если они менялись в течение года (два периода — две версии карточки).

### Что хранить
```sql
CREATE TABLE kpi_card_versions (
    id SERIAL PRIMARY KEY,
    pos_id INTEGER NOT NULL,
    role_id VARCHAR NOT NULL,
    version INTEGER NOT NULL,          -- 1, 2, 3...
    valid_from DATE NOT NULL,          -- с какого периода действует
    valid_to DATE,                     -- до какого (NULL = текущая)
    indicators JSONB NOT NULL,         -- полный снимок карточки
    created_by INTEGER,                -- redmine_id кто загрузил
    created_at TIMESTAMP,
    change_reason TEXT,                -- причина изменения
    status VARCHAR DEFAULT 'draft'     -- draft / active
);
```

### Привязка submission к версии карточки
```sql
-- В таблице kpi_submissions добавить:
kpi_card_version_id INTEGER REFERENCES kpi_card_versions(id)
```

При создании submission — фиксируем актуальную версию карточки.
При формировании PDF — используем именно эту версию.

## 5. ЗАГРУЗКА КАРТОЧЕК

### Вариант А — Полная загрузка (xlsx)
- Загрузить новый `KPI_Mapping.xlsx`
- Система показывает **diff**: какие карточки изменились, что добавилось, что удалилось
- Admin подтверждает → новые версии создаются для изменившихся карточек
- Неизменившиеся карточки не трогаются
- Новые версии получают статус `draft` до акцепта

### Вариант Б — Загрузка одной карточки (JSON/форма)
Формат для загрузки/редактирования одной карточки:

```json
{
  "pos_id": 75,
  "role_id": "ЦТР_КОН_075",
  "change_reason": "Уточнение показателей для новой функции",
  "indicators": [
    {
      "indicator": "Название показателя",
      "criterion": "Критерий оценки",
      "plan_value": "100%",
      "weight": 30,
      "is_common": false,
      "formula_type": "binary_auto",
      "formula_desc": "Описание",
      "thresholds": "Выполнено→100% | Не выполнено→0%",
      "cumulative": false
    }
  ]
}
```

- Загрузка через форму в admin-панели (поле за полем) **или** через JSON-файл
- Валидация перед сохранением: сумма весов = 100%, все обязательные поля заполнены
- После сохранения — статус `draft`, требует акцепта admin

### Акцепт изменений
1. HR загружает новую версию → статус `draft`
2. Admin видит карточки в статусе `draft` в интерфейсе
3. Admin проверяет diff (было → стало)
4. Admin акцептует → статус `active`, `valid_from` = начало следующего периода
5. Или отклоняет с комментарием → HR получает уведомление

## 6. ПРОВЕРКА КОРРЕКТНОСТИ КАРТОЧКИ

Автоматическая проверка при загрузке и в диагностике:

| Проверка | Условие валидности |
|---|---|
| Сумма весов | = 100% (±0.01 для округлений) |
| Обязательные поля | indicator, criterion, weight, formula_type заполнены |
| Тип формулы | только: binary_auto / binary_manual / threshold / multi_threshold / quarterly_threshold |
| Пороги | для threshold/multi_threshold/quarterly_threshold поле thresholds не пустое |
| Общие показатели | is_common=true показатели присутствуют (дисциплина, распорядок, охрана труда) |
| pos_id в системе | сотрудник с таким pos_id существует в employees |

## 7. ПРИВЯЗКА КАРТОЧКИ К СОТРУДНИКУ

### Автоматически (основной путь)
При синхронизации: `CF_211 (КПИ_Номер)` из Redmine → `pos_id` → актуальная версия карточки.

### При смене должности в Redmine
1. Синхронизация обнаруживает изменение CF_211
2. Система **уведомляет admin**: «Сотрудник X сменил должность. Проверьте карточку в Redmine»
3. Admin проверяет корректность карточки вручную в Redmine
4. **Исследовать:** можно ли обновить CF_211 через Redmine API автоматически?
   - `PUT /users/{id}.json` с обновлением custom_fields — проверить доступность

### Ручная корректировка
Если admin видит что сотруднику назначена неверная карточка (неверный pos_id):
- Admin исправляет CF_211 в Redmine вручную
- При следующей синхронизации — карточка обновится автоматически
- Прямого переопределения карточки в KPI-портале нет — Redmine источник истины

## 8. ИНТЕРФЕЙС

### Admin-панель /admin → таб «KPI-карточки»

#### Список карточек
- Таблица: pos_id / Должность / Подразделение / Показателей / Сумма весов / Статус / Версия
- Фильтр по подразделению
- Поиск по названию должности
- Статусы:
  - ✅ Валидна — сумма весов = 100%, все поля заполнены
  - ⚠️ Черновик — есть версия в статусе draft, ожидает акцепта
  - ❌ Невалидна — сумма весов ≠ 100% или пустые обязательные поля
  - 👤 Нет сотрудника — карточка есть, но сотрудника с таким pos_id нет

#### Просмотр карточки
- Клик на строку → детальный просмотр карточки
- Таблица показателей: Показатель / Критерий / Тип / Вес / Пороги / Нарастающий
- История версий: список версий с датами и причинами изменений
- Diff между версиями

#### Действия
- «Загрузить KPI_Mapping.xlsx» → полная загрузка с diff-предпросмотром
- «Добавить/изменить карточку» → форма для одной карточки (HR инициирует)
- «Акцептовать» / «Отклонить» → для draft-версий (только admin)

### Роль HR
- Видит все карточки (только просмотр актуальных)
- Может создать draft новой версии карточки
- Не может акцептовать — только admin
- Получает уведомление при отклонении admin

## 10. РЕЗУЛЬТАТЫ ИССЛЕДОВАНИЯ REDMINE API

### PUT /users/{id}.json — запись CF_211 ✅ РАБОТАЕТ

Проверено: `PUT /users/412.json` с `{"user": {"custom_fields": [{"id": 211, "value": "77"}]}}` — успешно.
Это означает: **автоматическое обновление должности (pos_id) через Redmine API возможно**.

### Что это меняет в архитектуре

| Сценарий | Было | Стало |
|---|---|---|
| Смена должности сотрудника | Admin вручную меняет CF_211 в Redmine | Портал может обновить CF_211 автоматически через API |
| Новый сотрудник без должности | Только уведомление admin | Портал предлагает выбрать pos_id и записывает в Redmine |
| Исправление ошибки карточки | Только через Redmine вручную | Admin выбирает pos_id в портале → API обновляет CF_211 |

### Новый флоу назначения должности

```
[Admin в портале]
  Видит сотрудника без pos_id (CF_211 пустой)
    ↓
  Выбирает должность из dropdown (все 91 из KPI_Mapping)
    ↓
  PUT /users/{redmine_id}.json → CF_211 = выбранный pos_id
    ↓
  Синхронизация подхватывает изменение при следующем запуске
```

### Ограничения
- API-ключ должен иметь права администратора Redmine для записи пользовательских полей
- Проверить: работает ли запись для любого пользователя или только для себя
