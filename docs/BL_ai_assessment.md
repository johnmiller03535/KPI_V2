## 1. ИСТОЧНИК ДАННЫХ ДЛЯ AI

### Что передаётся в YandexGPT
Система собирает `time_entries` из Redmine за отчётный период для конкретного сотрудника:

```
GET /time_entries.json?user_id={redmine_id}&from={period_start}&to={period_end}
```

Каждая запись содержит:
- `issue.subject` — название задачи
- `hours` — трудозатраты в часах
- `comments` — комментарий к записи
- `spent_on` — дата

### Что НЕ допускается
- AI не должен придумывать факты если данных нет
- Если `time_entries` пустой → текст: «Трудозатраты за период не зафиксированы в Redmine»
- AI не должен использовать общие фразы без опоры на реальные задачи

## 2. ФОРМАТ САММАРИ

### Требования к тексту
- Формальный деловой стиль
- Отвечает на вопрос: **что конкретно сделано**
- Перечисление выполненных работ через точку с запятой или новую строку
- Не более 3-5 предложений

### Пример правильного саммари
```
Проведён анализ критериев и показателей расчёта КПИ.
Подготовлено и настроено окружение Redmine для учёта трудозатрат.
Разработаны модули формирования отчётов KPI-портала.
Проведено тестирование сквозного цикла оценки.
```

### Пример неправильного саммари (текущее поведение)
```
Сотрудник выполнил широкий спектр задач, включая подготовку аналитических данных,
разработку модулей формирования отчётов, участие в совещаниях и защиту проектов.
Все задачи были выполнены в срок.
```
❌ Слишком общо, нет конкретики, похоже на шаблон

## 3. ПРОМПТ ДЛЯ YANDEXGPT

Передавать в API структурированно:

```
Ты — система оценки KPI государственного учреждения.
На основе трудозатрат сотрудника за период составь формальное саммари выполненных работ.

Показатель KPI: {indicator_name}
Критерий оценки: {criterion}
Период: {period_start} — {period_end}
Сотрудник: {employee_name}, должность: {role_name}

Трудозатраты из Redmine:
{time_entries_list}
(формат: "Задача: [название] | Часов: [N] | Дата: [дата]")

Требования к ответу:
1. Только факты из трудозатрат — не придумывай
2. Деловой стиль, 2-4 предложения
3. Начинай с глагола действия: "Проведён...", "Разработан...", "Выполнен..."
4. Если трудозатрат нет — ответь: "Трудозатраты за период не зафиксированы"
```

## 4. БЛОКИ ОЦЕНКИ — РАСПРЕДЕЛЕНИЕ KPI

### Правила распределения

| formula_type | is_common | Блок |
|---|---|---|
| binary_auto | false | ⚡ AI-оценка |
| binary_auto | true | ⚡ AI-оценка |
| binary_manual | false | 👤 Оценка руководителем |
| binary_manual | true | 👤 Оценка руководителем |
| threshold | any | 📊 Числовые показатели (сотрудник вводит факт) |
| multi_threshold | any | 📊 Числовые показатели |
| quarterly_threshold | any | 📊 Числовые показатели |

### Важно
**Все** `binary_manual` показатели — независимо от `is_common` — должны попасть в блок «Оценка руководителем».
`is_common=true` определяет только то, что показатель общий для всех должностей, но не меняет кто его оценивает.

### Известный баг
Один из общих показателей (`is_common=true`, `binary_manual`) попадает в блок AI-оценки вместо блока руководителя.
Причина: вероятно фильтрация идёт только по `is_common` без учёта `formula_type`.

**Правильная фильтрация:**
```python
ai_indicators = [i for i in indicators if i.formula_type == 'binary_auto']
manager_indicators = [i for i in indicators if i.formula_type == 'binary_manual']
numeric_indicators = [i for i in indicators if i.formula_type in ('threshold', 'multi_threshold', 'quarterly_threshold')]
```

## 6. ПЕРЕСМОТРЕННЫЙ ФЛОУ (подтверждено 2026-04-28)

### Было (неверно)
Сотрудник нажимает «Сгенерировать AI-оценку» → AI сразу оценивает → сотрудник отправляет

### Стало (правильно)

```
ШАГ 1 — Загрузка саммари
  Сотрудник нажимает «Загрузить саммари»
  Система берёт time_entries из Redmine и формирует текст-заготовку
  Текст показывается сотруднику в редактируемом поле

ШАГ 2 — Редактирование
  Сотрудник может отредактировать саммари вручную
  Уточнить формулировки, добавить контекст
  Поле сохраняется автоматически (debounce)

ШАГ 3 — Отправка отчёта
  Сотрудник нажимает «Отправить на проверку»
  В этот момент система вызывает YandexGPT:
    - Передаёт саммари (отредактированное сотрудником)
    - Передаёт критерии KPI
    - Получает оценку (Выполнено/Не выполнено) и уверенность

ШАГ 4 — Показ результата AI
  Сотрудник видит результат оценки AI
  Если уверенность < 50% → предупреждение:
    «⚠️ AI оценил с низкой уверенностью. Руководитель примет финальное решение»
  После этого форма становится read-only

ШАГ 5 — Руководитель
  Получает уведомление в Telegram
  Видит саммари + оценку AI + уверенность
  Принимает решение по binary_manual показателям
  Утверждает или отклоняет
```

### Что меняется в интерфейсе

**Было:** кнопка «⚡ Сгенерировать AI-оценку»
**Стало:** кнопка «📥 Загрузить саммари из Redmine»

**Было:** AI оценивает сразу при нажатии кнопки
**Стало:** AI оценивает в момент отправки отчёта

**Было:** сотрудник может перегенерировать AI-оценку
**Стало:** после отправки — форма read-only, AI-оценка финальна

### Хранение в БД

В таблице `kpi_submissions` добавить поля:
```sql
summary_text TEXT,           -- саммари введённое/отредактированное сотрудником
summary_loaded_at TIMESTAMP, -- когда загрузили из Redmine
submitted_at TIMESTAMP,      -- когда отправлено (момент AI-оценки)
```

В таблице `kpi_values` (оценки по каждому показателю):
```sql
ai_confidence FLOAT,         -- уверенность AI (0.0 — 1.0)
ai_reasoning TEXT,           -- обоснование AI
ai_low_confidence BOOLEAN,   -- флаг низкой уверенности (< 0.5)
```
