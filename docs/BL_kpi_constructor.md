# Бизнес-логика: Конструктор KPI показателей
> Версия: 1.0 | Дата: 2026-04-30 | Статус: подтверждено

---

## 1. МОДЕЛЬ ДАННЫХ

### Показатель (Indicator) — базовая сущность
```json
{
  "id": "uuid",
  "code": "IND_001",
  "name": "Осуществление закупок конкурентными способами",
  "formula_type": "threshold",
  "is_common": false,
  "is_editable_per_role": true,
  "status": "active",
  "created_by": "ZaichkoVV",
  "version": 3,
  "valid_from": "2026-04-01"
}
```

### Критерий показателя (Criterion) — конкретная формулировка
Один показатель может иметь несколько критериев (для разных должностей).
```json
{
  "id": "uuid",
  "indicator_id": "uuid",
  "criterion": "Сумма конкурентных закупок / общая сумма × 100%",
  "numerator_label": "Сумма конкурентных закупок (руб.)",
  "denominator_label": "Общая сумма закупок (руб.)",
  "thresholds": [
    {"condition": ">=67", "score": 100},
    {"condition": ">=50", "score": 50},
    {"condition": "<50", "score": 0}
  ],
  "cumulative": false,
  "plan_value": ">=67%"
}
```

### Карточка должности (RoleCard) — набор показателей для pos_id
```json
{
  "pos_id": 1,
  "role_id": "РУК_ПЕРЗ_001",
  "role_name": "Первый заместитель директора",
  "version": 2,
  "valid_from": "2026-04-01",
  "indicators": [
    {
      "indicator_id": "uuid",
      "criterion_id": "uuid",
      "weight": 20,
      "order": 1,
      "overrides": {}
    }
  ]
}
```

### Переопределение (Override) — индивидуальная правка для должности
Если общий показатель нужно незначительно изменить для конкретной должности — руководитель создаёт override. Но только для `is_editable_per_role = true` показателей.
```json
{
  "criterion": "уточнённая формулировка",
  "thresholds": [...],
  "weight": 25
}
```

## 2. ТИПЫ ПОКАЗАТЕЛЕЙ И ИХ КОНСТРУКТОР

### binary_auto
Оценивает AI на основе трудозатрат.
```
Поля: name, criterion, weight, is_common
```

### binary_manual
Оценивает руководитель (✅/❌).
```
Поля: name, criterion, weight, is_common
Стандартные формулировки: да/нет (для is_common)
```

### threshold
Числитель / знаменатель → % → пороги.
```
Поля: name, criterion, numerator_label, denominator_label, weight
Пороги: список {condition, score} — например ">=67%→100%"
cumulative: bool (нарастающим итогом)
```

### multi_threshold
Несколько подпоказателей, каждый со своей формулой.
Невыполнение одного = невыполнение всего.
```
Поля: name, weight
Подпоказатели[]: {label, numerator_label, denominator_label, threshold}
```

### quarterly_threshold
Пороги меняются по кварталам.
```
Поля: name, criterion, numerator_label, denominator_label, weight
Пороги по кварталам:
  Q1: {condition, score}
  Q2: {condition, score}
  Q3: {condition, score}
  Q4: {condition, score}
Логика квартала: по дате окончания отчётного периода
```

## 3. РОЛИ И ПРАВА

| Действие | Руководитель | HR | Admin |
|---|---|---|---|
| Просмотр всех показателей | ✅ | ✅ | ✅ |
| Создать новый показатель (draft) | ✅ | ✅ | ✅ |
| Редактировать is_common показатель | ❌ | ✅ (draft) | ✅ |
| Добавить показатель к своей должности | ✅ (draft) | ✅ | ✅ |
| Создать override для своей должности | ✅ (draft) | ✅ | ✅ |
| Утвердить изменения (draft→active) | ❌ | ❌ | ✅ |
| Вернуть на доработку | ❌ | ✅ | ✅ |
| Удалить показатель | ❌ | ❌ | ✅ |

### Флоу утверждения
```
Руководитель/HR создаёт → статус draft
HR проверяет → комментарий или передаёт admin
Admin утверждает → статус active, valid_from = начало следующего периода
```

---

## 4. ВЕРСИОНИРОВАНИЕ

- Каждое утверждение создаёт новую версию показателя/карточки
- Старые версии хранятся с `valid_to` датой
- `KpiSubmission` фиксирует версию карточки на момент создания
- Изменения вступают в силу со **следующего** отчётного периода
- Текущий активный период всегда работает со старой версией

---

## 5. ОБЩИЕ ПОКАЗАТЕЛИ (is_common)

- Существуют в единственном экземпляре
- При изменении → меняются у **всех** привязанных должностей одновременно
- Редактировать может только HR (создаёт draft) и Admin (утверждает)
- Руководитель не может создать override для is_common показателя
- Три стандартных общих показателя:
  1. Соблюдение исполнительской дисциплины (МСЭД) — 30%, binary_manual
  2. Соблюдение правил трудового распорядка — 10%, binary_manual
  3. Соблюдение техники безопасности — 10%, binary_manual

## 6. ИНТЕРФЕЙС КОНСТРУКТОРА

### Страница /admin → таб «KPI-показатели»

#### Библиотека показателей
- Таблица: Код / Название / Тип / Общий / Используется в N должностях / Статус
- Фильтр по типу, статусу, подразделению
- Поиск по названию
- Кнопка «+ Создать показатель»

#### Форма создания/редактирования показателя
Шаг 1 — Тип показателя (переключатель):
```
○ binary_auto  ○ binary_manual  ○ threshold  ○ multi_threshold  ○ quarterly_threshold
```

Шаг 2 — Основные поля (зависят от типа):

**Для binary_auto / binary_manual:**
```
Название показателя: [__________]
Критерий оценки:     [__________]
Общий показатель:    □
```

**Для threshold:**
```
Название показателя:   [__________]
Критерий оценки:       [__________]
Числитель:             [__________]  (например: «Кол-во конкурентных закупок»)
Знаменатель:           [__________]  (например: «Общее кол-во закупок»)
Нарастающим итогом:    □
Пороги:
  [условие] → [оценка %]    + добавить порог
  >= 67     → 100
  >= 50     → 50
  < 50      → 0
```

**Для multi_threshold:**
```
Название показателя: [__________]
Подпоказатели:
  [+ Добавить подпоказатель]
  ┌─────────────────────────────────────────┐
  │ Название: [__________]                  │
  │ Числитель: [__________]                 │
  │ Знаменатель: [__________]               │
  │ Порог: [условие] → [оценка]             │
  └─────────────────────────────────────────┘
Логика: все подпоказатели должны быть выполнены
```

**Для quarterly_threshold:**
```
Название показателя: [__________]
Критерий:            [__________]
Числитель:           [__________]
Знаменатель:         [__________]
Пороги по кварталам:
  Q1: [условие] → [оценка]
  Q2: [условие] → [оценка]
  Q3: [условие] → [оценка]
  Q4: [условие] → [оценка]
```

Шаг 3 — Предпросмотр JSON и кнопка «Сохранить как черновик»

#### Карточка должности
- Выбор должности из dropdown (все 91)
- Список привязанных показателей с весами
- Сумма весов = 100% (валидация)
- Кнопка «+ Добавить показатель» → выбор из библиотеки
- Кнопка «Изменить вес» для каждого показателя
- Кнопка «Удалить из карточки»
- Для is_common → только просмотр (нет кнопки удаления)

## 8. ПРИОРИТЕТ РЕАЛИЗАЦИИ

1. **БД** — таблицы `indicators`, `criteria`, `role_cards`, `role_card_indicators`
2. **API** — CRUD для показателей и карточек должностей
3. **UI** — таб «KPI-показатели» в admin-панели
   - Библиотека показателей
   - Конструктор форм по типу
   - Карточка должности
4. **Импорт** — парсинг текущего xlsx → заполнение новых таблиц
5. **Экспорт** — генерация xlsx из БД (замена текущего файла)

## 10. ОТВЕТЫ НА ОТКРЫТЫЕ ВОПРОСЫ (подтверждено 2026-04-30)

| # | Вопрос | Решение |
|---|---|---|
| 1 | Руководитель видит чьи карточки? | Свою должность + карточки всех подчинённых |
| 2 | Может ли руководитель создать новый показатель? | Да — функциональные обязанности могут меняться |
| 3 | Уведомлять сотрудников при изменении карточки? | Отложено — реализуем позже |
| 4 | Удаление показателя используемого в submissions? | Блокировать с сообщением: «Показатель используется в карточках: [список]. Сначала удалите из карточек» |

### Детали по пункту 1 — видимость карточек
```
Руководитель видит:
  - Карточку своей должности (pos_id)
  - Карточки всех подчинённых должностей (по матрице subordination)

Руководитель может:
  - Редактировать карточки подчинённых (создавать draft)
  - Предлагать новые показатели для подчинённых
  - НЕ может редактировать is_common показатели
```

### Детали по пункту 4 — защита от удаления
При попытке удалить показатель:
1. Проверить все `role_cards` где он используется
2. Если найдены → вернуть ошибку с перечнем карточек
3. Показать ссылки на каждую карточку для быстрого перехода
4. После удаления из всех карточек → разрешить удаление показателя
