## КОНТЕКСТ

Текущая форма «Новый показатель» — одна модалка для всех типов, JSON-поля, нет подсказок.
Нужно полностью переделать под 6 типов с визуальным редактором без JSON.

Бэкенд трогаем минимально: только миграция для нового типа `multi_binary`.
Вся логика конструктора — фронтенд.

---

## ЧАСТЬ 0 — Бэкенд: миграция и новый тип

### Новая миграция Alembic
```sql
ALTER TYPE formula_type ADD VALUE 'multi_binary';
ALTER TABLE kpi_criteria ADD COLUMN IF NOT EXISTS sub_type VARCHAR DEFAULT NULL;
ALTER TABLE kpi_criteria ADD COLUMN IF NOT EXISTS "order" INTEGER DEFAULT 0;
```

### API POST /api/admin/indicators/
Добавить поддержку `formula_type = 'multi_binary'` в схему и сохранение.
Подпоказатели (`sub_type = 'sub_binary'`) передаются в поле `criteria: list[{description, order}]`.

---

## ЧАСТЬ 1 — Исправление существующих багов

### БАГ: Поиск не работает
Фильтр `indicators.filter(i => i.name.toLowerCase().includes(query))` применять
к полному массиву `indicators` (все 124+), не к отфильтрованному по группе.
Поиск должен работать поверх активного фильтра группы ИЛИ сбрасывать фильтр группы.

### БАГ: После создания показатель не появляется
После успешного `POST`: закрыть модалку → вызвать `fetchIndicators()` → показатель виден в списке.

### БАГ: Группа vs Общий показатель
- Убрать «⭐ Общие показатели» из дропдауна ГРУППА — это не группа, а флаг
- Чекбокс переименовать: «Общий для всех сотрудников»
- Tooltip: «Показатель автоматически добавляется всем 91 сотруднику. Только для HR/admin»

## ЧАСТЬ 2 — Новая модалка: структура и навигация

### Размер и позиция
```css
.modal-overlay { position: fixed; inset: 0; display: flex;
  align-items: center; justify-content: center;
  background: rgba(0,0,0,0.75); z-index: 50; }
.modal-content { width: 700px; max-width: 95vw;
  max-height: 88vh; overflow-y: auto; }
```

### Шапка модалки
```
НОВЫЙ ПОКАЗАТЕЛЬ                                           ✕
────────────────────────────────────────────────────────────
ТИП *
[Выберите тип показателя ▼]

┌─────────────────────────────────────────────────────────┐
│ 💡 <описание выбранного типа одной строкой>             │
└─────────────────────────────────────────────────────────┘
```

Описания типов (показывать под select после выбора):
- `binary_auto` → «AI оценивает автоматически по трудозатратам сотрудника в Redmine»
- `binary_manual` → «Руководитель оценивает вручную: ✅ выполнено / ❌ не выполнено»
- `multi_binary` → «Руководитель оценивает каждый подпоказатель отдельно — нужно выполнить все»
- `threshold` → «Сотрудник вводит факт/план, система считает % и применяет пороги»
- `multi_threshold` → «Несколько числовых подпоказателей — нужно выполнить все»
- `quarterly_threshold` → «Числовой с разными порогами для каждого квартала»

### Общие поля (для всех типов)
```
НАЗВАНИЕ *
[textarea, 3 строки]
Подсказка: Полное официальное название показателя из методики

ГРУППА *                              ☐ Общий для всех сотрудников
[select: все группы кроме "Общие" ▼]  Tooltip: добавляется всем 91 сотруднику

КРИТЕРИЙ ОЦЕНКИ *
[textarea, 4 строки]
Подсказка: Опишите что именно оценивается и по какому принципу
```

## ЧАСТЬ 3 — Форма для каждого типа

### 3.1 binary_auto и binary_manual
Только общие поля (название + группа + критерий). Никаких дополнительных блоков.

---

### 3.2 multi_binary — Подпоказатели (бинарные)

После общих полей:
```
ПОДПОКАЗАТЕЛИ *
Подсказка: Каждый подпоказатель руководитель оценит отдельно ✅/❌.
           Невыполнение хотя бы одного = 0 баллов.

┌──────────────────────────────────────────────────────┐
│ 1. [Описание подпоказателя...                    ] 🗑 │
│ 2. [Описание подпоказателя...                    ] 🗑 │
│ 3. [Описание подпоказателя...                    ] 🗑 │
└──────────────────────────────────────────────────────┘
[+ Добавить подпоказатель]
```

- Минимум 2 подпоказателя (иначе используйте binary_manual)
- Каждый подпоказатель — однострочный input с кнопкой 🗑
- Порядок меняется перетаскиванием (drag handle ⠿) или стрелками ▲▼

---

### 3.3 threshold — Числовой с порогами

После общих полей:

**Блок «Числовой показатель»:**
```
ЧИСЛИТЕЛЬ *
[input]
Подсказка: Что считаем (факт). Пример: «Сумма конкурентных закупок (руб.)»

ЗНАМЕНАТЕЛЬ *
[input]
Подсказка: База для расчёта % (план). Пример: «Общая сумма закупок (руб.)»

☐ Нарастающим итогом
   Подсказка: Данные суммируются с начала года до конца текущего периода
```

**Блок «Пороги оценки»:**
```
ПРАВИЛА ОЦЕНКИ *
Подсказка: Условия проверяются сверху вниз. Первое совпавшее → балл.
           Обязательно добавьте правило с баллом 0 (минимум).

┌─────────────────────────────────────────────────┐
│  Условие              Балл                       │
│  [>= ▼] [ 95 ] %  →  [100]  🗑                  │
│  [>= ▼] [ 80 ] %  →  [ 50]  🗑                  │
│  [<  ▼] [ 80 ] %  →  [  0]  🗑                  │
└─────────────────────────────────────────────────┘
[+ Добавить правило]
```

Каждая строка:
- Select оператора: `>=` / `>` / `<=` / `<` / `=`
- Input числа (0–100, только цифры и точка)
- Символ `%` (фиксированный)
- `→`
- Input балла (0 / 50 / 100, или свободный ввод)
- Кнопка 🗑

По умолчанию при открытии формы — 2 пустые строки.

---

### 3.4 multi_threshold — Несколько числовых подпоказателей

После общих полей:

```
ПОДПОКАЗАТЕЛИ *
Подсказка: Каждый подпоказатель оценивается отдельно.
           Все должны быть выполнены (балл 100) — иначе итог 0.

┌──────────────────────────────────────────────────────────────┐
│ ПОДПОКАЗАТЕЛЬ 1                                         🗑   │
│ Название: [Конкуренция (количество участников)         ]     │
│ Числитель: [Кол-во участников конкурентных процедур    ]     │
│ Знаменатель: [Кол-во конкурентных процедур             ]     │
│ ☐ Нарастающим итогом                                         │
│ Правила: [>= ▼][2.7]% → [100] 🗑  [< ▼][2.7]% → [0] 🗑     │
│          [+ Добавить правило]                                 │
├──────────────────────────────────────────────────────────────┤
│ ПОДПОКАЗАТЕЛЬ 2                                         🗑   │
│ ...                                                          │
└──────────────────────────────────────────────────────────────┘
[+ Добавить подпоказатель]
```

Минимум 2 подпоказателя.

---

### 3.5 quarterly_threshold — Квартальные пороги

После числителя / знаменателя / нарастающим итогом (те же поля что threshold):

```
ПРАВИЛА ПО КВАРТАЛАМ *
Подсказка: Каждый квартал — отдельный набор правил.
           Квартал определяется по дате окончания периода.

┌─[Q1]──[Q2]──[Q3]──[Q4]────────────────────────────────┐
│                                                         │
│  Условие              Балл                              │
│  [>= ▼] [ 15 ] %  →  [100]  🗑                         │
│  [<  ▼] [ 15 ] %  →  [  0]  🗑                         │
│  [+ Добавить правило]                                   │
│                                                         │
│  [Скопировать Q1 во все кварталы →]                     │
└─────────────────────────────────────────────────────────┘
```

- 4 таба: Q1, Q2, Q3, Q4
- Каждый таб — независимая таблица правил (та же структура что в threshold)
- Кнопка «Скопировать Q1 во все кварталы» — копирует правила Q1 в Q2, Q3, Q4
- По умолчанию все 4 таба пустые (2 строки-заготовки)

## ЧАСТЬ 4 — Валидация

При нажатии «+ Создать показатель» — проверить:

| Тип | Условие | Ошибка под полем |
|---|---|---|
| Все | Название пустое | «Введите название показателя» |
| Все | Критерий пустой | «Опишите критерий оценки» |
| Все | Группа не выбрана | «Выберите группу» |
| multi_binary | Менее 2 подпоказателей | «Добавьте хотя бы 2 подпоказателя» |
| multi_binary | Есть пустые описания | «Заполните описание подпоказателя» |
| threshold | Числитель пустой | «Укажите что считается в числителе» |
| threshold | Знаменатель пустой | «Укажите базу для расчёта %» |
| threshold | Нет ни одного правила | «Добавьте хотя бы одно правило оценки» |
| threshold | Нет правила с баллом 0 | «Добавьте правило для минимального балла (0)» |
| threshold | Значение не число | «Введите число от 0 до 100» |
| multi_threshold | Менее 2 подпоказателей | «Добавьте хотя бы 2 числовых подпоказателя» |
| quarterly_threshold | Хотя бы 1 таб без правил | «Заполните правила для всех кварталов (Q1-Q4)» |

Ошибки — красный текст под соответствующим полем (`color: var(--danger)`).

## ЧАСТЬ 5 — Редактирование существующего показателя

Кнопка «Редактировать» в списке открывает ту же форму, но с заполненными данными.
Для `multi_binary` — подпоказатели загружаются из `kpi_criteria` (sub_type = 'sub_binary').
Для `quarterly_threshold` — правила раскладываются по табам Q1-Q4.
