## ОБЩИЕ ПРИНЦИПЫ

Каждый показатель имеет:
- **Название** — полное описание что оценивается
- **Критерий оценки** — как определяется выполнение
- **Вес** — доля в итоговой оценке (сумма по карточке = 100%)
- **Тип** — определяет КТО оценивает и КАК считается балл

Итоговый балл показателя: **0**, **50** или **100**.
Итоговая оценка сотрудника = сумма (балл × вес) по всем показателям.

## ТИП 1: binary_auto — Автоматическая оценка AI

### Кто оценивает
YandexGPT. Анализирует трудозатраты из Redmine (поле `comments` в time_entries).

### Когда применяется
Для показателей, результат которых виден из описания работ сотрудника.
Примеры: «Обеспечение проектной деятельности», «Обеспечение бизнес-анализа»

### Как считается балл
AI читает саммари трудозатрат → **100** (выполнено) или **0** (не выполнено).
Руководитель может переопределить (`manager_override`).

### Поля при создании
| Поле | Пример |
|---|---|
| Название | «Обеспечение проектной деятельности» |
| Критерий оценки | «Участие в проектах, подготовка проектной документации» |

Числитель / знаменатель / пороги — **не нужны**.

---

## ТИП 2: binary_manual — Ручная оценка руководителя

### Кто оценивает
Руководитель. В форме проверки нажимает ✅ или ❌ — одно решение на весь показатель.

### Когда применяется
Для показателей, требующих экспертной оценки без числовых данных.
Примеры: «Соблюдение трудового распорядка», «Соблюдение исполнительской дисциплины»

### Как считается балл
✅ → **100** баллов | ❌ → **0** баллов

### Поля при создании
| Поле | Пример |
|---|---|
| Название | «Соблюдение Правил трудового распорядка, Кодекса этики» |
| Критерий оценки | «Отсутствие нарушений трудового распорядка и Кодекса этики» |

Числитель / знаменатель / пороги — **не нужны**.

---

## ТИП 3: multi_binary — Составной бинарный (несколько подпоказателей)

### Кто оценивает
Руководитель. Оценивает **каждый подпоказатель отдельно**: ✅ или ❌.

### Когда применяется
Когда показатель включает несколько самостоятельных условий (не числовых),
каждое из которых должно быть выполнено.
Невыполнение хотя бы одного = 0 баллов для всего показателя.

Примеры из методики:
- «Составление планов и контроль выполнения»:
  - Для каждого проекта разработан план реализации
  - Осуществляется контроль выполнения плана
- «Контроль реализации бизнес-процессов»:
  - Проведён анализ и моделирование процессов
  - Процессы ранжированы по степени важности
  - Соблюдены сроки предоставления отчётности

### Как считается балл
```
Все подпоказатели ✅  →  100 баллов
Хотя бы один ❌       →  0 баллов
```

### Отличие от binary_manual
`binary_manual` — одно решение руководителя.
`multi_binary` — N отдельных решений, каждое со своим описанием, итог один.

### Хранение в БД
Каждый подпоказатель — отдельная строка в `kpi_criteria`:
```
indicator_id  → ссылка на показатель
description   → «Для каждого проекта разработан план реализации»
order         → 1, 2, 3...
sub_type      → 'sub_binary'
```

### Поля при создании
| Поле | Пример |
|---|---|
| Название | «Контроль реализации моделирования бизнес-процессов» |
| Критерий оценки | «Все условия должны быть соблюдены в полном объёме» |
| Подпоказатель 1 | «Проведён анализ и моделирование процессов» |
| Подпоказатель 2 | «Процессы ранжированы по степени важности» |
| [+ Добавить подпоказатель] | кнопка добавляет строку |

## ТИП 4: threshold — Числовой с едиными порогами

### Кто оценивает
Сотрудник вводит числитель и знаменатель. Система считает % и применяет пороги.

### Когда применяется
Для показателей с чёткой формулой: факт / план × 100%.
Примеры: «Объём конкурентных закупок ≥67%», «Доля жалоб <1%»

### Как считается балл
```
% = числитель / знаменатель × 100
→ правила проверяются сверху вниз
→ первое совпавшее условие → балл
```

### Структура правил
Список: оператор (`>=`, `>`, `<=`, `<`, `=`) + значение + балл (0 / 50 / 100).

Пример (три уровня):
```
>= 95% → 100
>= 80% → 50
<  80% → 0
```

Пример (бинарный):
```
>= 67% → 100
<  67% → 0
```

### Поля при создании
| Поле | Пример |
|---|---|
| Название | «Объём конкурентных закупок» |
| Критерий оценки | «Сумма конкурентных закупок / общая сумма закупок × 100%» |
| Числитель | «Сумма закупок, осуществлённых конкурентным способом (руб.)» |
| Знаменатель | «Общая сумма закупок заказчика (руб.)» |
| Нарастающим итогом | ✅ да / ☐ нет |
| Правила порогов | таблица (строки добавляются через «+ Добавить правило») |

## ТИП 5: multi_threshold — Несколько числовых подпоказателей

### Кто оценивает
Сотрудник вводит данные по каждому подпоказателю отдельно.

### Когда применяется
Когда показатель = несколько числовых условий, все обязательны.
Пример: «Конкуренция ≥2,7 И объём закупок ≥67% И жалоб <1%» — три независимых числа.

### Как считается балл
```
Для каждого подпоказателя: % = числитель_i / знаменатель_i × 100 → балл_i
Все балл_i = 100  →  100 баллов итого
Хотя бы один < 100  →  0 баллов итого
```

### Хранение в БД
Каждый подпоказатель — строка в `kpi_criteria` со своим числителем, знаменателем, правилами, `sub_type = 'sub_numeric'`.

### Поля при создании
Для каждого подпоказателя (добавляются через «+ Добавить подпоказатель»):
| Поле | Пример |
|---|---|
| Название подпоказателя | «Конкуренция (количество участников)» |
| Числитель | «Количество участников конкурентных процедур» |
| Знаменатель | «Количество конкурентных процедур» |
| Правила порогов | >= 2.7 → 100; < 2.7 → 0 |

## ТИП 6: quarterly_threshold — Числовой с квартальными порогами

### Кто оценивает
Сотрудник вводит числитель и знаменатель.
Система определяет квартал периода и применяет соответствующие правила.

### Когда применяется
Для показателей где требование меняется по кварталам — нарастающий план.
Примеры:
- «Объём СМП нарастающим итогом: Q1≥15%, Q2≥25%, Q3≥35%, Q4≥45%»
- «Доля единственного поставщика: Q1<10%, Q2<20%, Q3<30%, Q4<40%»

### Ключевой принцип: каждый квартал независим
Каждый квартал имеет свой набор правил — как отдельный `threshold`.
Оператор, значения и баллы могут отличаться между кварталами.
Система не накладывает ограничений на логику — всё через конструктор.

**Пример А — «не менее» (растущий нижний порог):**
```
Q1: >= 15% → 100,  <  15% → 0
Q2: >= 25% → 100,  <  25% → 0
Q3: >= 35% → 100,  <  35% → 0
Q4: >= 45% → 100,  <  45% → 0
```

**Пример Б — «не более» (растущий верхний порог, инвертированная логика):**
```
Q1: <  10% → 100,  >= 10% → 0
Q2: <  20% → 100,  >= 20% → 0
Q3: <  30% → 100,  >= 30% → 0
Q4: <  40% → 100,  >= 40% → 0
```

**Пример В — смешанная (разная логика по кварталам):**
```
Q1: >= 80% → 100,  >= 60% → 50,  < 60% → 0
Q2: >= 85% → 100,  >= 65% → 50,  < 65% → 0
Q3: <   5% → 100,  >=  5% → 0               ← другой оператор
Q4: >= 90% → 100,  <  90% → 0
```

### Определение квартала по дате окончания периода
```
Январь   — Март     → Q1
Апрель   — Июнь     → Q2
Июль     — Сентябрь → Q3
Октябрь  — Декабрь  → Q4
Весь год (годовой)  → Q4
```

### Хранение в БД (kpi_criteria.thresholds как JSON)
```json
{
  "Q1": [{"operator": ">=", "value": 15, "score": 100}, {"operator": "<", "value": 15, "score": 0}],
  "Q2": [{"operator": ">=", "value": 25, "score": 100}, {"operator": "<", "value": 25, "score": 0}],
  "Q3": [{"operator": ">=", "value": 35, "score": 100}, {"operator": "<", "value": 35, "score": 0}],
  "Q4": [{"operator": ">=", "value": 45, "score": 100}, {"operator": "<", "value": 45, "score": 0}]
}
```

### Поля при создании
- Числитель, знаменатель, нарастающим итогом — как в threshold
- Пороги: **4 таба Q1 / Q2 / Q3 / Q4**
- В каждом табе — таблица правил (оператор + значение % + балл)
- Кнопка **«Скопировать Q1 во все кварталы»** — для быстрого старта
- Каждый таб независим — можно задать любую логику

## СВОДНАЯ ТАБЛИЦА

| Тип | Кто оценивает | Вводит данные | Балл | Подпоказатели | Пороги |
|---|---|---|---|---|---|
| binary_auto | AI | — | 0 / 100 | ❌ | ❌ |
| binary_manual | Руководитель | Руководитель (✅/❌) | 0 / 100 | ❌ | ❌ |
| multi_binary | Руководитель | Руководитель (✅/❌ каждый) | 0 / 100 | ✅ с описанием | ❌ |
| threshold | Система | Сотрудник (числитель + знаменатель) | 0 / 50 / 100 | ❌ | ✅ единые |
| multi_threshold | Система | Сотрудник (по каждому подпоказателю) | 0 / 100 | ✅ числовые | ✅ для каждого |
| quarterly_threshold | Система | Сотрудник (числитель + знаменатель) | 0 / 50 / 100 | ❌ | ✅ по Q1-Q4 |

## ПОЛЕ «НАРАСТАЮЩИМ ИТОГОМ»

Только для `threshold`, `multi_threshold`, `quarterly_threshold`.

**ДА** — сотрудник вводит данные накопленным итогом с начала года.
Пример: в марте вводит сумму за январь + февраль + март.

**НЕТ** — данные только за текущий отчётный период.

## ОБЩИЕ ПОКАЗАТЕЛИ (is_common = true)

Три показателя у всех 91 сотрудника — нельзя убрать из карточки:

| Показатель | Тип | Вес |
|---|---|---|
| Исполнительская дисциплина (МСЭД, поручения) | binary_manual | 30% |
| Соблюдение трудового распорядка, Кодекса этики | binary_manual | 10% |
| Соблюдение охраны труда и противопожарного режима | binary_manual | 10% |

Итого общие = 50%. Оставшиеся 50% — специфические показатели должности.

`is_common` — это **флаг**, не группа в сайдбаре.
Устанавливает только HR/admin через конструктор.

## ИЗМЕНЕНИЯ В БД (новый тип multi_binary)

Требуется новая миграция Alembic:

```sql
-- 1. Добавить новое значение в enum formula_type
ALTER TYPE formula_type ADD VALUE 'multi_binary';

-- 2. Добавить поле sub_type в kpi_criteria (уже существует)
ALTER TABLE kpi_criteria ADD COLUMN sub_type VARCHAR DEFAULT NULL;
-- sub_type = 'sub_binary'  — подпоказатель multi_binary
-- sub_type = 'sub_numeric' — подпоказатель multi_threshold
-- sub_type = NULL          — обычный критерий (threshold, binary_*)
```
