# BL_indicator_types.md — Бизнес-логика типов KPI показателей

> Версия: 3.0 | Дата: 2026-05-04 | Статус: финал
> Изменения v3.0: добавлен тип absolute_threshold (7-й тип)

---

## ОБЩИЕ ПРИНЦИПЫ

Каждый показатель имеет:
- **Название** — полное описание что оценивается
- **Критерий оценки** — как определяется выполнение
- **Вес** — доля в итоговой оценке (сумма по карточке = 100%)
- **Тип** — определяет КТО оценивает и КАК считается балл

Итоговый балл показателя: **0**, **50** или **100**.
Итоговая оценка сотрудника = сумма (балл × вес) по всем показателям.

---

## СВОДНАЯ ТАБЛИЦА (7 типов)

| Тип | Кто оценивает | Что вводит сотрудник | Формула | Пороги |
|---|---|---|---|---|
| binary_auto | AI | — | — | ❌ |
| binary_manual | Руководитель | — | — | ❌ |
| multi_binary | Руководитель | — | — | ❌ |
| threshold | Система | числитель + знаменатель | числ/знам×100% | ✅ единые |
| multi_threshold | Система | несколько пар числ/знам | каждая пара отдельно | ✅ для каждого |
| quarterly_threshold | Система | числитель + знаменатель | числ/знам×100% | ✅ Q1-Q4 |
| **absolute_threshold** | **Система** | **одно число** | **прямое сравнение** | **✅ единые или Q1-Q4** |

---

## ТИП 1: binary_auto — Автоматическая оценка AI

### Кто оценивает
YandexGPT. Анализирует трудозатраты из Redmine (поле `comments`).

### Как считается балл
AI → **100** (выполнено) или **0** (не выполнено). Руководитель может переопределить.

### Поля при создании
- Название, Группа, Критерий оценки

---

## ТИП 2: binary_manual — Ручная оценка руководителя

### Кто оценивает
Руководитель: одно решение ✅/❌ на весь показатель.

### Как считается балл
✅ → **100** | ❌ → **0**

### Поля при создании
- Название, Группа, Критерий оценки

---

## ТИП 3: multi_binary — Составной бинарный

### Кто оценивает
Руководитель: оценивает каждый подпоказатель отдельно ✅/❌.

### Как считается балл
Все ✅ → **100** | Хотя бы один ❌ → **0**

### Поля при создании
- Название, Группа, Критерий оценки
- Подпоказатели (минимум 2, каждый с описанием)

### Хранение в БД
Подпоказатели — строки в `kpi_criteria` с `sub_type = 'sub_binary'`.

---

## ТИП 4: threshold — Числовой с процентными порогами

### Кто оценивает
Сотрудник вводит числитель и знаменатель. Система считает % и применяет пороги.

### Как считается балл
```
% = числитель / знаменатель × 100
→ правила проверяются сверху вниз → первое совпавшее → балл
```

### Поля при создании
- Название, Группа, Критерий оценки
- Числитель (что считаем — факт)
- Знаменатель (база для % — план)
- Нарастающим итогом (да/нет)
- Правила порогов: оператор + значение% + балл

---

## ТИП 5: multi_threshold — Несколько числовых подпоказателей

### Кто оценивает
Сотрудник вводит числитель/знаменатель для каждого подпоказателя.

### Как считается балл
```
Для каждого подпоказателя: % = числ_i / знам_i × 100 → балл_i
Все балл_i = 100 → итог 100 | Хотя бы один < 100 → итог 0
```

### Поля при создании
- Название, Группа, Критерий оценки
- Подпоказатели (минимум 2): каждый имеет числитель, знаменатель, правила порогов

---

## ТИП 6: quarterly_threshold — Числовой с квартальными порогами

### Кто оценивает
Сотрудник вводит числитель и знаменатель.
Система определяет квартал по дате окончания периода и применяет правила квартала.

### Определение квартала
```
Январь–Март        → Q1
Апрель–Июнь        → Q2
Июль–Сентябрь      → Q3
Октябрь–Декабрь    → Q4
Весь год (годовой) → Q4
```

### Как считается балл
```
% = числитель / знаменатель × 100
→ применяем правила текущего квартала → балл
```

Каждый квартал — независимый набор правил (как отдельный threshold).

### Поля при создании
- Название, Группа, Критерий оценки
- Числитель, Знаменатель, Нарастающим итогом
- 4 таба Q1/Q2/Q3/Q4 — в каждом таблица правил
- Кнопка «Скопировать активный квартал во все»

### Хранение в БД (kpi_criteria.thresholds)
```json
{
  "Q1": [{"operator": ">=", "value": 15, "score": 100}, {"operator": "<", "value": 15, "score": 0}],
  "Q2": [{"operator": ">=", "value": 25, "score": 100}, {"operator": "<", "value": 25, "score": 0}],
  "Q3": [{"operator": ">=", "value": 35, "score": 100}, {"operator": "<", "value": 35, "score": 0}],
  "Q4": [{"operator": ">=", "value": 45, "score": 100}, {"operator": "<", "value": 45, "score": 0}]
}
```

---

## ТИП 7: absolute_threshold — Абсолютный числовой показатель ← НОВЫЙ

### Суть отличия от threshold
`threshold` — сотрудник вводит **два** числа (числитель и знаменатель), система считает **%**.
`absolute_threshold` — сотрудник вводит **одно** число, система сравнивает его с порогами **напрямую** (без деления).

### Когда применяется
Для показателей где результат — штуки, количество, уровень — а не доля от плана.

Примеры из методики:
- «Правовая экспертиза локальных актов» — количество повторных согласований (≤2 → 100)
- «Оптимизация выпуска документов» — количество изданных актов (≥3 → 100)
- «Подготовка аналитических материалов» — количество материалов в месяц (>16 → 100)
- «Подготовка информационных материалов» — количество материалов в месяц (>22 → 100)
- «Количество участников на торгах» — среднее количество участников (квартальные пороги)

### Два режима: единые пороги и квартальные

**is_quarterly = false** (единые пороги):
```
Сотрудник вводит: [число]
Система сравнивает с правилами → балл

Пример: <=2 → 100 | >2 → 0
```

**is_quarterly = true** (квартальные пороги):
```
Сотрудник вводит: [число]
Система определяет квартал по дате периода → применяет правила квартала → балл

В месячном отчёте: квартал = квартал этого месяца
В квартальном:    квартал = этот квартал
В годовом:        квартал = Q4

Пример Q1: >=2.7 → 100 | <2.7 → 0
Пример Q2: >=3.0 → 100 | <3.0 → 0
```

### Кто оценивает
Сотрудник вводит значение вручную при заполнении KPI-формы.

### Как считается балл
```
value = введённое сотрудником число
→ если is_quarterly: взять правила текущего квартала
→ правила проверяются сверху вниз → первое совпавшее → балл
```

### Поля при создании (конструктор)
- Название, Группа, Критерий оценки
- Подпись поля ввода (`value_label`): например «Количество материалов» или «Среднее число участников»
- Нарастающим итогом (да/нет)
- Чекбокс **«Квартальные пороги»**:
  - false → одна таблица правил (оператор + значение + балл)
  - true → 4 таба Q1/Q2/Q3/Q4

### Хранение в БД
```sql
-- kpi_indicators
formula_type = 'absolute_threshold'
-- kpi_criteria
value_label = 'Количество материалов'  ← новое поле
is_quarterly = true/false
thresholds = [{"operator": ">=", "value": 3, "score": 100}, ...]
-- или для квартальных:
thresholds = {"Q1": [...], "Q2": [...], "Q3": [...], "Q4": [...]}
```

## ПОКАЗАТЕЛИ ТРЕБУЮЩИЕ ИЗМЕНЕНИЯ ТИПА (2026-05-04)

### Перевести из threshold → binary_manual (нет числовых порогов)
| Показатель | Причина |
|---|---|
| Оптимизация локальных актов Учреждения | Нет порогов в xlsx |
| Своевременная подготовка заключений по НПА | Нет порогов в xlsx |
| Соблюдение сроков подготовки ответов | Нет порогов в xlsx |
| Обеспечение моделирования бизнес-процессов | Пороги = «Соблюдается/Не соблюдается» |

### Перевести из threshold/quarterly_threshold → absolute_threshold
| Показатель | is_quarterly | Пороги |
|---|---|---|
| Правовая экспертиза проектов локальных актов | false | `<=2→100 / >2→0` |
| Оптимизация выпуска документов | false | `>=3→100 / <3→0` |
| Подготовка аналитических материалов по ЗИТ | false | `>16→100 / =16→50 / <16→0` |
| Подготовка информационных материалов по ЗИТ | false | `>22→100 / =22→50 / <22→0` |
| Количество участников на торгах по МО | true | Q1:>=2.7 / Q2:>=3.0 / Q3:>=3.3 / Q4:>=3.6 → 100 |

## ПОЛЕ «НАРАСТАЮЩИМ ИТОГОМ»

Для `threshold`, `multi_threshold`, `quarterly_threshold`, `absolute_threshold`.

**ДА** — сотрудник вводит данные накопленным итогом с начала года.
**НЕТ** — данные только за текущий период.

---

## ОБЩИЕ ПОКАЗАТЕЛИ (is_common = true)

Три показателя у всех 91 сотрудника — нельзя убрать из карточки:

| Показатель | Тип | Вес |
|---|---|---|
| Исполнительская дисциплина (МСЭД, поручения) | binary_manual | 30% |
| Соблюдение трудового распорядка, Кодекса этики | binary_manual | 10% |
| Соблюдение охраны труда и противопожарного режима | binary_manual | 10% |

## ИЗМЕНЕНИЯ В БД (шаг K-prep)

```sql
-- 1. Новый тип в enum
ALTER TYPE formula_type ADD VALUE 'absolute_threshold';

-- 2. Новые поля в kpi_criteria
ALTER TABLE kpi_criteria ADD COLUMN IF NOT EXISTS value_label VARCHAR DEFAULT NULL;
-- value_label: подпись поля ввода для absolute_threshold
-- is_quarterly уже должно быть (или добавить если нет)
ALTER TABLE kpi_criteria ADD COLUMN IF NOT EXISTS is_quarterly BOOLEAN DEFAULT FALSE;

-- 3. Обновить 4 показателя на binary_manual
UPDATE kpi_indicators SET formula_type = 'binary_manual'
WHERE name IN (
  'Оптимизация локальных актов Учреждения',
  'Своевременная подготовка заключений по проектам нормативных правовых актов...',
  'Соблюдение, установленных законодательством Российской Федерации сроков...',
  'Обеспечение моделирования бизнес-процессов'
);

-- 4. Обновить 5 показателей на absolute_threshold
UPDATE kpi_indicators SET formula_type = 'absolute_threshold'
WHERE name IN (
  'Правовая экспертиза проектов локальных актов Учреждения',
  'Оптимизация выпуска документов, регламентирующих деятельность Учреждения',
  'Подготовка аналитических материалов по ЗИТ',
  'Подготовка информационных материалов по популяризации ЗИТ',
  'Количество участников на торгах по Московской области'
);
```
