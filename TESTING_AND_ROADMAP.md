## БИЗНЕС-ПРОЦЕСС СИСТЕМЫ

### Участники
- **Администратор** — настраивает систему, создаёт периоды
- **Сотрудник** — заполняет KPI-отчёт за период
- **Руководитель** — проверяет и утверждает отчёты подчинённых
- **Финансовый блок** — просматривает утверждённые отчёты

### Полный цикл отчётного периода

```
[АДМИНИСТРАТОР]
    │
    ▼
1. Настройка подчинённости (кто кому подчиняется)
    │
    ▼
2. Создание отчётного периода (месяц/квартал/год)
    │
    ▼
3. Создание задач в Redmine для всех сотрудников
    │
    ▼ (автоматически создаются KPI-submission для каждого)

[СОТРУДНИК]
    │
    ▼
4. Списание трудозатрат в Redmine в течение периода
   (комментарии к записям — основной источник для AI)
    │
    ▼
5. Открытие KPI-формы → нажатие "Загрузить трудозатраты" → редактирование саммари
    │
    ▼
6. "Сгенерировать AI-оценку":
   - binary_auto KPI → AI ставит оценку + summary
   - numeric KPI → сотрудник вводит факт вручную
   - binary_manual KPI → ожидают оценки руководителя
    │
    ▼
7. Просмотр результатов → отправка на проверку

[РУКОВОДИТЕЛЬ]
    │
    ▼
8. Получение уведомления в Telegram
    │
    ▼
9. Открытие формы проверки:
   - Просмотр саммари сотрудника + AI-оценок (binary_auto) с возможностью override
   - Просмотр числовых показателей (numeric)
   - Ручная оценка общих показателей (binary_manual): ✅/❌ + стандартная формулировка
    │
    ▼
10. Утверждение или отклонение отчёта
   - При отклонении → сотрудник получает уведомление + причину
   - При утверждении → генерируется PDF, прикрепляется к Redmine

[ФИНАНСОВЫЙ БЛОК]
    │
    ▼
11. Просмотр утверждённых отчётов + PDF
    Сводка по подразделениям
```

## ЧТО РЕАЛИЗОВАНО

### Инфраструктура и данные
- Синхронизация сотрудников из Redmine (81 чел., 9 подразделений)
- Импорт матрицы подчинённости из People export (Excel → `subordination.json`)
- `GET /api/admin/data-health` — состояние данных: без должности, без Telegram, без подчинённых
- Таб «Подчинённость» в admin-панели — просмотр дерева иерархии

### Периоды и задачи
- CRUD периодов (draft → active → review → closed)
- Создание KPI-задач в Redmine (dry_run поддержан)
- Автоматическое создание KPI-submission при создании задач
- Исключения сотрудников из периода (dismissed/transferred/maternity)

### KPI-форма сотрудника
- Новый флоу AI-оценки: `load-summary` → редактирование текста → `generate-summary` → `submit`
- Трудозатраты Redmine обогащаются subject задачи (asyncio.gather + отдельные API-запросы)
- Классификация: рабочие / отпускные записи (по `project.name` и `activity.name`)
- AI-саммари использует поле `comments` как основной источник данных

### Проверка руководителем
- `manager_override` для binary_auto показателей — переопределяет AI-оценку
- binary_manual показатели: кнопки ✅/❌ без перезагрузки страницы
- Стандартные формулировки для `is_common` показателей (словарь `COMMON_KPI_TEXTS`)
- Блокировка «Утвердить» при наличии непроставленных ручных оценок
- `computeScore` учитывает `manager_override` везде: dashboard, список /review, detail

### PDF-отчёт
- Период в формате `дд.мм.гггг — дд.мм.гггг` (из `period.date_start/date_end`)
- Подпись: должность руководителя из KPI_Mapping + ФИО из таблицы `employees`
- Колонка «Мероприятия»: `summary_text` сотрудника (не AI reasoning)
- Стандартные формулировки для общих показателей в таблице отчёта
- Страница «Детализация показателей» удалена

### Дизайн и UX
- Dark Cyber дизайн-система на всех страницах (Orbitron, Exo 2, glassmorphism)
- `src/lib/kpiScore.ts` — единый модуль расчёта KPI-оценки
