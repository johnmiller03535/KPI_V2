## БИЗНЕС-ПРОЦЕСС СИСТЕМЫ

### Участники
- **Администратор** — настраивает систему, создаёт периоды
- **Начальник управления / Зам. директора** — создаёт новые KPI-показатели (draft)
- **HR** — проверяет новые показатели (review) ← роль не реализована
- **Сотрудник** — заполняет KPI-отчёт за период
- **Руководитель** — проверяет и утверждает отчёты подчинённых
- **Финансовый блок** — просматривает утверждённые отчёты

### Полный цикл отчётного периода

```
[АДМИНИСТРАТОР]
    │
    ▼
1. Проверить структуру: Redmine → «Импорт из People» → ⚠️ должности без карточек
    │
    ▼
2. Убедиться что у всех должностей есть KPI-карточки (все 91 ✅)
    │
    ▼
3. Создание отчётного периода → задачи в Redmine

[СОТРУДНИК]
    │
    ▼
4. Списание трудозатрат в Redmine в течение периода
    │
    ▼
5. KPI-форма → «Загрузить саммари» → AI оценивает binary_auto
   Числовые показатели → сотрудник вводит числитель/знаменатель  ← шаг K
    │
    ▼
6. Отправить на проверку

[РУКОВОДИТЕЛЬ]
    │
    ▼
7. Уведомление в Telegram
    │
    ▼
8. multi_binary и binary_manual → ✅/❌ → утверждение/отклонение

[ФИНАНСОВЫЙ БЛОК]
    │
    ▼
9. Просмотр утверждённых отчётов + PDF
```

## ПЛАН ТЕСТИРОВАНИЯ ШАГА K

После реализации числовых KPI:

- [ ] threshold: поля числитель/знаменатель видны для threshold-показателей
- [ ] threshold: ввод данных → % считается правильно (числитель/знаменатель×100)
- [ ] threshold: пороги применяются сверху вниз → правильный балл
- [ ] quarterly_threshold: определяется правильный квартал по дате окончания периода
- [ ] quarterly_threshold: Q1 (янв-март), Q2 (апр-июн), Q3 (июл-сен), Q4 (окт-дек / год)
- [ ] quarterly_threshold: применяются пороги текущего квартала
- [ ] multi_threshold: отдельные поля для каждого подпоказателя
- [ ] multi_threshold: все выполнены → 100, хотя бы один не выполнен → 0
- [ ] multi_binary: руководитель видит каждый подпоказатель отдельно
- [ ] multi_binary: все ✅ → 100, хотя бы одна ❌ → 0
- [ ] Нарастающим итогом: подсказка для сотрудника что вводить

## ОТКРЫТЫЕ ЗАДАЧИ (бэклог)

### Workflow утверждения показателей
```
draft → (HR проверяет в UI) → review → (Admin утверждает в UI) → active
```
UI для этого workflow не реализован. Сейчас показатели создаются как `draft` и остаются там.

### Роль HR
Права: проверять новые показатели (draft→review), менять `is_common`, создавать карточки.

### Activity types → KPI маппинг (шаг I)
Виды деятельности Redmine привязываются к KPI-показателям для точного AI-саммари.
В `KPI_Mapping.xlsx` добавить колонку activity_names.

### Тестовые показатели
При тестировании конструктора создано несколько тестовых показателей (draft).
Нужно пометить их `is_active=false` или удалить из БД напрямую.
